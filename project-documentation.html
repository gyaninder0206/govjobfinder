1. Project Overview

The app is a full-stack job discovery product focused on government jobs. The frontend is a React single-page app. The backend exposes authentication, jobs, and saved-jobs APIs. Logged-in users can browse jobs and maintain a saved-jobs list in MongoDB.

What the project currently does

  • Shows a marketing-style landing page.
  • Lets users sign up and log in.
  • Keeps users logged in with a cookie-based JWT.
  • Stores and updates user profile data in MongoDB.
  • Fetches government jobs from a third-party API through the backend.
  • Normalizes external job data into a simpler internal shape.
  • Lets authenticated users save and unsave jobs.
  • Displays saved jobs on a protected route.
  • Uses environment-based frontend/backend configuration for deployment.

What it does not fully implement yet

  • No real resume upload API exists yet.
  • No personalized recommendation engine exists yet.
  • No advanced eligibility-matching logic exists yet.
  • No roles, admin features, or notifications backend exist yet.

3. Frontend Routes and Pages

The frontend route system is defined in frontend/src/App.jsx using Routes, Route, and conditional rendering based on the user state.

Route Page / Component Access What it does
/ Home Public Shows the landing page with hero, features, pricing, FAQ, CTA, and optional profile/resume modals.
/jobs Jobs Protected in frontend If user exists, fetches jobs from the backend and allows save/unsave actions. Otherwise redirects to /.
/saved-jobs SavedJobs Protected in frontend Shows the logged-in user's saved jobs. Otherwise redirects to /.

Route protection logic

  • The app stores the current user in top-level state inside App.
  • On app load, it calls /api/auth/me with credentials: "include".
  • If that call succeeds, the returned user becomes the active session user.
  • If it fails, the app treats the visitor as logged out.
  • /jobs and /saved-jobs render only when user is truthy.

Page-level behavior summary

  • Home: marketing UI + modal entry points.
  • Jobs: job fetch + save heart toggle.
  • Saved Jobs: saved list + remove heart toggle + fallback empty state.

4. Frontend Components and Their Logic

Below is how the major UI pieces work, including real behavior behind the buttons and modals.

App.jsx

  • Owns the top-level user state.
  • Owns the top-level savedJobs state.
  • Checks the current session on first render using /api/auth/me.
  • Loads saved jobs after login using fetchSavedJobs().
  • Defines toggleSavedJob(job) which either saves or removes a job.
  • Wraps the entire UI with background visuals, navbar, router, and footer.

src/utils/api.js

  • Centralizes the frontend API base URL.
  • Reads VITE_API_URL when present.
  • Falls back to http://localhost:5000 for local development.
  • Prevents hardcoded production URLs from being scattered across components.

Navbar.jsx

  • Shows sign-in action when logged out.
  • Shows initials-based user menu when logged in.
  • Lets the user open the saved-jobs page.
  • Logs the user out by calling POST /api/auth/logout.
  • Uses local component state for mobile menu, login modal, profile dropdown, and toast messages.
  • Closes the profile menu when the user clicks outside of it.
  • Uses the shared API URL helper instead of a hardcoded localhost backend.

LoginModal.jsx

  • Handles both sign up and sign in inside one modal.
  • In signup mode it sends name, email, and password to /api/auth/signup.
  • In login mode it sends email and password to /api/auth/login with cookies enabled.
  • After login it immediately calls /api/auth/me to fetch the user object and update app state.
  • Displays success and error messages directly in the modal.
  • Uses the shared API URL helper, making it ready for hosted environments.

GetStartedModal.jsx

  • Available only after the user is logged in.
  • Offers two actions: create or update profile, and upload resume.
  • Contains no backend communication itself.
  • Opens nested modal components for each action.
  • Closes on ESC key or backdrop click.

CreateProfileForm.jsx

  • Captures name, age, gender, qualification, category, state, email, and phone.
  • Prefills the form from the logged-in user if a profile already exists.
  • Stores in-progress values in React state while editing.
  • Submits the form to PUT /api/auth/profile with credentials included.
  • Updates the live authenticated user state after save so the UI reflects profile changes immediately.
  • Supports both first-time profile creation and later profile updates in the same modal.

UploadResumeModal.jsx

  • Lets the user pick a file locally.
  • Does not upload the real file to the server.
  • Stores file metadata only in localStorage under userResume.
  • Shows a temporary success state, then closes automatically.

CTA.jsx

  • Checks the authenticated user's real Mongo-backed profile plus the locally stored resume metadata.
  • If no backend profile exists, it opens the create-profile flow.
  • If profile exists but resume metadata does not, it opens resume upload.
  • If both exist, it navigates to jobs.
  • This means the CTA now relies on actual saved profile data instead of a client-only placeholder key.

Jobs.jsx and SavedJobs.jsx

  • Jobs fetches job listings from GET /api/jobs.
  • It compares each listed job with the saved-jobs array using isSameJob().
  • The heart button toggles save/unsave behavior through a callback from App.
  • SavedJobs displays the user's current saved jobs from state and removes them with the same callback.
  • The jobs and saved-jobs helpers now use a centralized environment-aware API base.

5. Backend Routes and APIs

The backend server is defined in backend/app.js. It mounts three route groups under /api/auth, /api/jobs, and /api/saved-jobs.

Method Route Protected Purpose Main Logic
GET / No Health-style root endpoint Returns simple text: Jobs API running.
POST /api/auth/signup No Create user account Checks if email exists, hashes password with bcrypt, inserts user into MongoDB.
POST /api/auth/login No Log user in Validates credentials, signs JWT, sets token HTTP-only cookie using environment-aware cookie options, and returns basic user info.
POST /api/auth/logout No Log user out Clears the token cookie.
GET /api/auth/me Yes Get current authenticated user Uses JWT from cookie, loads user from DB, removes password from response, and returns the user's profile if present.
PUT /api/auth/profile Yes Create or update profile Validates profile fields, updates name and email, writes the nested profile object to MongoDB, and returns the updated user.
GET /api/jobs No Fetch government jobs Gets jobs from cache or RapidAPI, normalizes fields, returns simplified array.
GET /api/saved-jobs Yes Get saved jobs Returns req.user.savedJobs.
POST /api/saved-jobs Yes Save a job Validates title and link, prevents duplicates, prepends the job, saves the user document.
DELETE /api/saved-jobs Yes Remove a saved job Filters matching job out of the user's saved-jobs array and saves the user document.

6. Middleware and Authentication Logic

The backend uses both global middleware and one custom route-protection middleware.

Global middleware in backend/app.js

  • express.json() parses incoming JSON bodies.
  • bodyparser.urlencoded({ extended: true }) parses URL-encoded form bodies.
  • cookieParser() makes cookies available at req.cookies.
  • cors(...) uses a whitelist derived from FRONTEND_URL and supports credentials.
  • The server listens on process.env.PORT || 5000, which makes it compatible with hosted Node platforms such as Render.

Custom auth middleware: requireAuth

  • Reads req.cookies.token.
  • If there is no token, returns 401 Not authenticated.
  • Verifies the token using JWT_SECRET.
  • Loads the user by decoded ID from MongoDB.
  • If the user is missing, returns 401 User not found.
  • On success attaches the Mongoose user document to req.user.
Login: frontend posts credentials to /api/auth/login.
Backend validates the password and signs a JWT containing the user ID.
Backend sets the JWT as an HTTP-only cookie named token.
Cookie settings switch by environment: local uses sameSite: "lax", production uses secure: true and sameSite: "none".
Frontend later calls protected endpoints with credentials: "include".
requireAuth verifies the cookie and reconstructs the logged-in user.

7. Database Model

The app currently has one MongoDB model: User.

User schema fields

  • name: required string
  • email: required unique string
  • password: required hashed string
  • profile: optional embedded profile object
  • savedJobs: array of embedded objects

Profile object fields:

  • age: number
  • gender: string
  • qualification: string
  • category: string
  • state: string
  • phone: string

Each saved job object contains:

  • title: required string
  • lastDate: string, default "Not specified"
  • applyLink: required string
  • savedAt: date, default current time

Schema timestamps: enabled, so each user also has createdAt and updatedAt.

8. End-to-End Data Flow

This section explains how data moves through the system for the core user journeys.

Sign up flow

User opens LoginModal and switches to signup mode.
Frontend sends name, email, password to POST /api/auth/signup.
Backend checks whether the email already exists.
Backend hashes the password with bcrypt and inserts the user.
Frontend shows success message and asks the user to log in manually.

Login flow

User submits email and password in login mode.
Backend validates user existence and password correctness.
Backend creates a 7-day JWT and sets the cookie.
Frontend then requests GET /api/auth/me.
Returned user object is stored in the top-level React state.

Session restore on page load

App.jsx runs a startup useEffect.
It calls GET /api/auth/me with cookies included.
If valid, the app restores the full user, including profile data, and then loads saved jobs.
If invalid, the app clears the user state and treats the visitor as logged out.

Profile save / update flow

Signed-in user opens Get Started and chooses create or update profile.
CreateProfileForm preloads existing user and profile values if present.
Frontend submits the form to PUT /api/auth/profile with cookies included.
Backend validates input, updates the user document, and stores the nested profile object in MongoDB.
Frontend replaces the in-memory user state with the returned updated user object.

Jobs fetch flow

Jobs.jsx fetches GET /api/jobs.
Backend calls getCachedJobs(fetchExternalJobs).
If cache is fresh, cached data is used; otherwise RapidAPI is called.
Raw external data is normalized into { title, lastDate, applyLink }.
Frontend renders cards and uses saved-jobs state to mark saved items.

Save job flow

User clicks the heart icon on a job card.
App.toggleSavedJob checks whether the job already exists in local state.
If not saved, frontend calls POST /api/saved-jobs with the job object.
Backend validates, deduplicates, prepends the job, saves the user document, and returns the updated list.
Frontend replaces savedJobs state with the returned array.

Remove saved job flow

User clicks the heart icon on an already-saved job.
Frontend calls DELETE /api/saved-jobs with the job object in the request body.
Backend filters that job out of req.user.savedJobs.
Backend saves the user document and returns the updated list.
Frontend updates savedJobs state from the response.

9. Feature-by-Feature Logic

This is the logic behind each important visible feature in the current codebase.

Landing page

  • Built from Hero, Features, Pricing, Faq, and CTA.
  • Most content here is static marketing content from component files and dummy-data.jsx.
  • Framer Motion animates entrance effects for sections and cards.
  • The landing page now also acts as the entry point for real profile creation and updates for signed-in users.

Deployment readiness

  • The frontend now uses VITE_API_URL instead of hardcoded API URLs.
  • The backend now supports configurable frontend origins via FRONTEND_URL.
  • The backend includes a production-friendly start script.
  • The frontend build script is now vite build, which matches the current repo setup better than the previous TypeScript-based command.

Smooth scrolling and visual effects

  • LenisScroll installs Lenis and drives it using requestAnimationFrame.
  • SoftBackdrop renders fixed blurred gradient blobs behind the UI.
  • index.css imports Tailwind and the Outfit font, plus custom marquee and utility classes.

Saved jobs identity logic

  • Both frontend and backend identify a job primarily by applyLink.
  • Fallback identity is title-lastDate if needed.
  • This shared logic helps keep save/unsave behavior consistent.

Job caching logic

  • The backend keeps jobs in in-memory variables: cachedJobs and lastFetchTime.
  • Cache duration is 1 hour.
  • This reduces external API calls while the server process stays alive.
  • Important limitation: the cache resets when the backend restarts.

Normalization logic

  • The external API response is expected to have a data array.
  • Each raw job is converted to only three frontend-facing fields.
  • This keeps the frontend simple and decouples it from the third-party response shape.

Linting setup

  • The frontend uses ESLint with React Hooks and React Refresh rules.
  • The noisy base no-unused-vars rule has been disabled in the current config because it was falsely flagging JSX-used imports.
  • The remaining lint pass is clean apart from a non-blocking package freshness notice.

Logout logic

  • Navbar calls POST /api/auth/logout.
  • Backend clears the auth cookie.
  • Frontend clears local user state, closes menus, shows a toast, and navigates home.

10. Current Gaps, Risks, and Important Notes

These are key implementation realities that matter if someone is onboarding to the project or extending it.

Profile persistence is now real, but resume and recommendation features are still partial. The application now stores user profile data in MongoDB, but resume upload is still local-only and personalized recommendations are not yet generated from profile data.

Gaps in current functionality

  • Profile data is stored, but it is not yet used to filter or rank jobs.
  • UploadResumeModal stores only file metadata locally, not the actual file.
  • There is no recommendation engine, despite UI text suggesting personalization.
  • Pricing plans are display-only and have no payment or subscription logic.
  • The backend does not yet persist enriched job metadata beyond the three normalized fields returned from the external API.

Security and production notes

  • The JWT cookie behavior is now environment-aware: local development uses relaxed settings, while production uses secure cross-site cookie settings.
  • CORS is now driven by FRONTEND_URL and can accept multiple comma-separated origins.
  • There is little request validation beyond presence checks.
  • There is no rate limiting, no CSRF mitigation layer, and no centralized error handling yet.

Environment variables used

# Backend PORT=5000 FRONTEND_URL=http://localhost:5173 MONGO_URI=your_mongodb_connection_string JWT_SECRET=your_jwt_secret RAPIDAPI_KEY=your_rapidapi_key RAPIDAPI_HOST=your_rapidapi_host # Frontend VITE_API_URL=http://localhost:5000

Deployment notes

  • The project is now structured to support a split deployment such as Vercel for the frontend and Render for the backend.
  • In production, the frontend should use the hosted backend URL through VITE_API_URL.
  • The backend should use the hosted frontend URL in FRONTEND_URL so credentialed requests are accepted by CORS.
  • MongoDB Atlas is the natural production database choice for this codebase.

Practical summary

The current project is best described as a strong starter full-stack app with working authentication, job fetching, saved-jobs persistence, MongoDB-backed profile creation and editing, and environment-based deployment configuration. The marketing copy still promises more than the implementation currently delivers around resume intelligence and personalized recommendations, but the core auth, profile, save-job, and deployment-ready runtime flows are now real.
